# Haiku → Strudel 하네스 품질 — 아이디어 스케치

> 상태: **아이디어 단계**. 미착수.
> 구현이 확정되면 실제 프롬프트는 `docs/harness/`(버전 관리)로, 여기엔 전략·근거만 남긴다.

## 문제 (사용자 아이디어)

Haiku의 "자연어 → Strudel 음악 언어" 변환 프롬프트가 가장 중요한 레버다.
현재 출력은 음악이라고 보기 어려운 수준이다. 해결하려면:
- 고품질 시스템 프롬프트
- Strudel 유저들이 많이 쓰는 기법, 샘플 프리셋 등의 정보

## 사용자가 제시한 드럼 규칙 (4마디 드럼 루프 기준)

이건 작곡가인 사용자의 도메인 지식이며, v3 프롬프트의 씨앗이다. **원문 보존**:

1. **스네어 드럼**은 반드시 한 마디의 2번째 박자, 4번째 박자마다 연주되어야 한다.
2. **킥 드럼**은 1–2마디의 패턴이 3–4마디와 유사하면 듣기 좋으며, 첫 번째 마디
   첫 번째 박자에는 반드시 등장해야 하고, 스네어 박자와 동시에 나오는 것을
   가급적 피하는 것이 좋다. (단, 4-beat 하우스 킥 드럼은 예외)
3. **closed hi-hat**은 한 마디에 4개 또는 8개가 연주되는 것을 기본으로 하며,
   4번째 마디의 3·4번째 박자에는 16분음표로 리듬 변형이 생길 수 있다.
4. **crash cymbal**은 8마디 단위로 반복되며, 첫 번째 마디의 첫 번째 박자에만 연주된다.

## Claude 제안

### 1. 레이어 타입별 규칙 초안 — bass / chord / melody
(드럼 규칙은 위 "사용자가 제시한 드럼 규칙" 참고)

#### bass (사용자 규칙 — 강도 순 (1) > (2) > (3), (1)이 최강)
1. **[최강] chord의 root음을 기반으로** 연주한다.
2. 리듬은 **킥과 비슷**해야 듣기 좋다.
3. 사용 음 우선순위: 옥타브음(root ±12) → 코드의 5음 → 코드음 →
   key의 diatonic 음 → 그 외 반음. 대부분 **root & root±12 & 5th** 까지만 써도 충분.

→ Claude 메모:
- 규칙 (1)을 지키려면 하네스가 **chord 레이어의 화성을 알아야** 한다. 현재
  `existingLayers`엔 코드 *문자열*만 넘어감 — 코드 진행을 구조화 데이터로 함께
  넘기는 게 안전 (Q2 참고).
- 규칙 (2) + 드럼 규칙(킥은 스네어와 동시 회피)이 맞물려 베이스가 그루브의 축.

#### chord (사용자 — (1)(2)는 아이디어 / (3)(4)는 규칙)
아이디어:
1. 최소 12개 Major/Minor key를 세팅해 둔다.
2. Money chords / 유명 진행 / 대중음악 빈출 진행(예: IIm–V7–IM)을 검색해 **프리셋**으로.

규칙:
3. 4마디 루프 = 한 마디에 코드 하나 → **코드 4개**가 일반 구성.
   (2마디 루프면 마디당 코드 둘)
4. 코드 변화는 마디 기준이되, **킥 리듬에 따라 마디보다 앞/뒤로** 밀고 당길 수 있다.

→ Claude 메모:
- (1)(2)는 프롬프트 규칙이 아니라 **데이터 자산** — 12키 × 진행 프리셋 테이블.
  하네스가 이 테이블에서 고르게 하거나 생성을 이 테이블로 제약. 별도 작업 항목.
- (3)은 Strudel `<>`로 깔끔히 매핑:
  `note("<[c3,e3,g3] [a2,c3,e3] [f2,a2,c3] [g2,b2,d3]>")`.
- (4) 앵커 밀고 당기기(anticipation/push)는 미니 노테이션으로 가능하나 고급 — v3 후속.

#### melody (사용자 규칙 — "가장 주관적, 어법이 가장 어려움")
1. 기본적으로 **key에 종속**되는 음. **diatonic 7음**을 쓰는 게 자연스럽다.

→ Claude 제안 (멜로디는 사용자가 "가장 주관적"이라 명시 → 작곡가 검토 필수):
- 프레이즈 구조: 2+2마디 call & response부터 시작하면 안전.
- 강박(1·3박)에 **코드톤** 착지 → 화성 충돌 감소. 약박엔 경과음 허용.
- 쉼표로 숨 쉬기 — 음을 꽉 채우지 않기. 음역 4~5옥타브(베이스와 분리).
- 위는 전부 *제안*. 멜로디는 규칙화보다 **few-shot GOLD 예시**에 더 기대는 게 맞을 듯.

### 2. 조성·화성 일관성 — `key`와 코드 진행을 thread
지금 첫 레이어가 bpm을 정한다. **key(조성)와 코드 진행도 위로 끌어올려** 정하고,
이후 레이어 응답에 함께 넘겨 모든 음(note) 레이어가 같은 조성·화성 위에 있게 한다.
- bass 규칙 (1)("chord의 root음 기반")이 성립하려면 bass 생성 시점에 코드 진행이
  *구조화된 데이터*로 있어야 한다 — 코드 문자열 파싱에 의존하면 깨지기 쉽다.
- 하네스 출력/입력 스키마에 `key` + `chordProgression` 필드 추가 검토. (Q2 참고)

### 3. Strudel 관용구 라이브러리를 프롬프트에 주입
하네스가 좋은 결과를 내는 Strudel 기법을 알아야 한다. 특히:
- 유클리드 리듬 `(3,8)`, `(5,8)` — 적은 코드로 그루브 있는 패턴. **레버리지 최상**
- `<a b>` 마디별 교대, `[a b]` 묶기, `~` 쉼표, `a*2` 반복
- `.slow()/.fast()`, `.every(n, f)`, `.gain()`, `.room()`
- 드럼 샘플 선택 `s("bd:3 sd:2")`

### 4. few-shot이 규칙보다 강하다 — GOLD 예시 병행
모델은 규칙 나열보다 좋은 예시를 더 잘 모방한다. 레이어 타입별로
**완성된 좋은 4마디 루프 예시 3~5개**를 넣고, 규칙은 그 예시를 뒷받침하는
형태로. (현재 하네스 v2에도 예시가 있으나 수가 적고 품질 기준이 낮음)

### 5. 장르 프리셋
학생이 vibe를 고르게 한다 — 힙합 / 하우스 / lo-fi / 팝 발라드.
장르마다 규칙팩을 둔다. 사용자의 드럼 규칙도 이미 장르 인식적이다
("4-beat 하우스 킥은 예외"). 규칙을 장르 축으로 정리하면 자연스럽다.

### 6. 프롬프트 평가 루프 — 개선을 측정 가능하게
"좋아졌다"를 감으로 판단하지 않으려면:
- 학생이 쓸 법한 입력 테스트셋 (20~30개)
- 출력 Strudel을 한 화면에서 빠르게 청취·비교하는 dev 페이지
프롬프트를 고칠 때마다 같은 셋을 돌려 A/B 청취.

### 7. 음악적 검증 (과투자 경계)
`isStrudelCodeSafe`는 위험 토큰만 막는다. 추가로 — 파싱 성공 여부,
드럼 레이어에 다운비트 킥 존재 여부 정도의 가벼운 체크 + 실패 시 1회 재생성.
단 일회성 수업 앱이므로 정교한 음악 분석까지는 과투자.

## 열린 질문

- **Q1. ✅ 결정됨 — 루프는 4마디.** crash 규칙은 "8마디 단위, 1마디 1박만"이므로
  4마디 루프에선 *루프 2회에 1번* crash가 나오거나, 단순화해 "루프 첫 마디 첫 박"으로
  본다. → 제안: v3에선 단순화("루프 첫 박") 채택, 8마디 구조는 후속.
- **Q2. 생성 순서 / 화성 데이터 threading.** bass 규칙 (1)은 코드 진행 의존,
  chord는 key 의존 → key·코드 진행이 bass/melody보다 **먼저** 정해져야 한다.
  현재는 첫 레이어가 bpm만 정함. 첫 레이어(또는 사전 스텝)가 key + 코드 진행도
  정하도록 아키텍처 변경 검토. (§2와 연결)
- **Q3. melody 규칙 보강 필요.** bass·chord는 사용자가 규칙을 제시함.
  melody는 아직 (1)뿐 — 작곡가 추가 입력 또는 few-shot GOLD 예시로 보완.
- **Q4.** 규칙을 시스템 프롬프트에 직접 서술 vs few-shot 예시로 녹임 vs 둘 다.
  (현 가설: 둘 다 — 규칙으로 경계를 긋고 예시로 품질 기준을 보임)
- **Q5.** 드럼 패드(`drum-pad.md`)의 그리드와 하네스 드럼 규칙은 같은 개념을
  공유 — 둘이 같은 규칙 정의를 단일 출처로 쓰게 할지.
