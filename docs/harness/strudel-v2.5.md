# Strudel 하네스 — v2.5

> 실제 프롬프트는 `apps/api/src/harness/strudel.ts`의 `STRUDEL_SYSTEM_PROMPT`.
> 이 문서는 버전 이력·근거. 설계 출처: `docs/ideas/harness-music-quality.md`.

## v2.5의 범위

**프롬프트 전용 개선** — 출력 스키마(`{code, explanation, bpm?}`)도, 입력
(`{layerType, description, bpm?, existingLayers[]}`)도, 클라이언트 조립 흐름도
그대로. 아키텍처 변경(Foundation 사전 스텝, key/chordProgression threading,
stateful chord 편집)은 v3로 미룸 — v2.5 테스트 결과를 보고 v3 착수 여부를 판단한다.

## v2 → v2.5 변경점

1. **레이어별 규칙 도입.** v2는 "타입에 맞는 패턴" 정도였음. v2.5는 작곡가(사용자)가
   제시한 규칙을 레이어별로 명시:
   - drum: 스네어 2·4박 / 킥 1마디 1박·스네어 동시 회피 / 하이햇 4·8개 / crash 루프 첫 박
   - bass: (1) chord 근음 기반 [최강] (2) 킥 리듬 모방 (3) 음 우선순위(근음·5음 위주)
   - chord: 4마디=코드 4개, 화성 기능(T/SD/D), 교육과정 용어(으뜸/버금딸림/딸림=I/IV/V),
     큐레이션 진행 5종
   - melody: 조성 종속 + 16분음표 그리드 + C3~C5 + 쉼표
2. **4마디 루프 표현 명시.** "한 Strudel 사이클 = 한 마디"이므로 4마디 루프는
   `<[1마디] [2마디] [3마디] [4마디]>`. v2엔 이 설명이 없어 1마디만 반복되곤 했음.
3. **문법 정정.** v2 예시의 `note("<c3,e3,g3> <a2,c3,e3>>")`는 문법 오류였음
   (`>>`). 올바른 순차 화음: `note("<[c3,e3,g3] [a2,c3,e3]>")`.
4. **GOLD 예시 5종**으로 교체 — 드럼(기본/4마디 변형), 코드, 베이스, 멜로디.
   이스케이프된 JSON 대신 읽기 쉬운 형식.
5. 유클리드 리듬 `(n,k)` 등 Strudel 관용구를 문법 섹션에 추가.

## v2.5의 알려진 한계 (v3에서 다룰 것)

- bass의 "chord 근음 기반"이 구조화 데이터가 아니라 **기존 레이어 code 문자열 파싱**에
  의존 — Haiku가 문자열에서 근음을 읽어야 함. 깨지기 쉬움.
- key가 스키마에 없어 **조성 일관성**이 "기존 레이어를 보고 맞춰라"는 지시에만 의존.
- chord 입력이 대화형 per-bar 편집이 아니라 한 번에 4코드 생성.
- crash 8마디 주기는 4마디 루프에 맞춰 "루프 첫 박"으로 단순화.

## 테스트 후 판단할 것

v2.5로 출력 품질이 "음악"이라 부를 만해졌는지. 부족하면 v3(아키텍처 변경)로,
충분하면 v2.x 미세 조정으로.
