# Haiku → Strudel 하네스 품질 — 아이디어 스케치

> 상태: **아이디어 단계**. 미착수.
> 구현이 확정되면 실제 프롬프트는 `docs/harness/`(버전 관리)로, 여기엔 전략·근거만 남긴다.

## 문제 (사용자 아이디어)

Haiku의 "자연어 → Strudel 음악 언어" 변환 프롬프트가 가장 중요한 레버다.
현재 출력은 음악이라고 보기 어려운 수준이다. 해결하려면:
- 고품질 시스템 프롬프트
- Strudel 유저들이 많이 쓰는 기법, 샘플 프리셋 등의 정보

## 사용자가 제시한 드럼 규칙 (4마디 드럼 루프 기준)

이건 작곡가인 사용자의 도메인 지식이며, v3 프롬프트의 씨앗이다. **원문 보존**:

1. **스네어 드럼**은 반드시 한 마디의 2번째 박자, 4번째 박자마다 연주되어야 한다.
2. **킥 드럼**은 1–2마디의 패턴이 3–4마디와 유사하면 듣기 좋으며, 첫 번째 마디
   첫 번째 박자에는 반드시 등장해야 하고, 스네어 박자와 동시에 나오는 것을
   가급적 피하는 것이 좋다. (단, 4-beat 하우스 킥 드럼은 예외)
3. **closed hi-hat**은 한 마디에 4개 또는 8개가 연주되는 것을 기본으로 하며,
   4번째 마디의 3·4번째 박자에는 16분음표로 리듬 변형이 생길 수 있다.
4. **crash cymbal**은 8마디 단위로 반복되며, 첫 번째 마디의 첫 번째 박자에만 연주된다.

## Claude 제안

### 1. 레이어 타입별 규칙 초안 — bass / chord / melody
(드럼 규칙은 위 "사용자가 제시한 드럼 규칙" 참고)

#### bass (사용자 규칙 — 강도 순 (1) > (2) > (3), (1)이 최강)
1. **[최강] chord의 root음을 기반으로** 연주한다.
2. 리듬은 **킥과 비슷**해야 듣기 좋다.
3. 사용 음 우선순위: 옥타브음(root ±12) → 코드의 5음 → 코드음 →
   key의 diatonic 음 → 그 외 반음. 대부분 **root & root±12 & 5th** 까지만 써도 충분.

→ Claude 메모:
- 규칙 (1)을 지키려면 하네스가 **chord 레이어의 화성을 알아야** 한다. 현재
  `existingLayers`엔 코드 *문자열*만 넘어감 — 코드 진행을 구조화 데이터로 함께
  넘기는 게 안전 (Q2 참고).
- 규칙 (2) + 드럼 규칙(킥은 스네어와 동시 회피)이 맞물려 베이스가 그루브의 축.

#### chord (사용자 — (1)(2)는 아이디어 / (3)(4)는 규칙)
아이디어:
1. 최소 12개 Major/Minor key를 세팅해 둔다.
2. Money chords / 유명 진행 / 대중음악 빈출 진행(예: IIm–V7–IM)을 검색해 **프리셋**으로.

규칙:
3. 4마디 루프 = 한 마디에 코드 하나 → **코드 4개**가 일반 구성.
   (2마디 루프면 마디당 코드 둘)
4. 코드 변화는 마디 기준이되, **킥 리듬에 따라 마디보다 앞/뒤로** 밀고 당길 수 있다.

→ Claude 메모:
- (1)(2)는 프롬프트 규칙이 아니라 **데이터 자산** — 12키 × 진행 프리셋 테이블.
  하네스가 이 테이블에서 고르게 하거나 생성을 이 테이블로 제약. 별도 작업 항목.
- (3)은 Strudel `<>`로 깔끔히 매핑:
  `note("<[c3,e3,g3] [a2,c3,e3] [f2,a2,c3] [g2,b2,d3]>")`.
- (4) 앵커 밀고 당기기(anticipation/push)는 미니 노테이션으로 가능하나 고급 — v3 후속.

#### 화성 기능 (사용자 제시) — 코드 진행 선택의 근거
Major Key 기준 7개 diatonic chord의 기능:
- **Tonic (T)**: I · III · VI
- **Sub-Dominant (SD)**: II · IV
- **Dominant (D)**: V · VII

화음은 기능에 따라 유기적으로 움직이며 **절대적인 규칙은 없다.** 다만:
1. **SD–D–T** 진행이 가장 강한 화성 진행감을 준다.
2. **root음이 5도 하행**하는 진행이 강력하게 들린다.
3. 그래서 **II–V–I** 같은 진행이 화성을 가장 기능적으로 잘 풀어낸 화음이다.

모든 음악이 II–V–I로 만들어지진 않는다. IV–V–VI (SD–D–T),
II–V–III–VI (SD–D–T–T) 같은 진행도 큰 기능 안에서 나올 수 있다.

> 더 정밀하게는 화성학 논문이나 관련 소프트웨어의 Git을 참조하는 게 best지만,
> 현재 단계에선 그 정도 분량의 구현은 불필요.

→ Claude 메모:
- 이 기능 이론은 Q2 「코드 진행 큐레이션」의 *근거*다. 큐레이션 진행을 기능으로 읽으면
  ① T–D–T–SD ② T–T–SD–D ③ T–SD–T–D ④ T–T–SD–D (⑤는 단조라 별도).
- 루프에서는 마지막 코드가 다시 첫 코드로 감긴다 → **D로 끝나는 ②③④**는 루프
  이음새에서 D–T 해결감을 얻고, SD로 끝나는 ①은 plagal(약한) 느낌.
- 하네스엔 "기능 라벨 + SD–D–T·5도 하행 선호"만 넣으면 충분. 전체 화성학은 과투자.

#### 학교 교육과정 연계 — 자연어 → 코드 해석 (사용자 제시)
결국 아이들이 학교에서 배우는 어휘와 연결되어야 한다. 장조 기준:
- **I = 으뜸화음** (Tonic)
- **IV = 버금딸림화음** (Sub-Dominant)
- **V = 딸림화음** (Dominant)

아이들에게 줄 응용 설명: **버금딸림화음 → 딸림화음**으로 가는 게 듣기 좋고,
**딸림화음 → 으뜸화음**으로 진행하는 게 좋다. (위 「화성 기능」의 SD–D–T를
교육과정 용어로 풀어쓴 것)

학생은 이 어휘로 자연어를 입력한다. Strudel chord 입력에서 해석해야 할 예:

| 학생 입력 | 해석 |
|-----------|------|
| 버금딸림화음을 첫 번째 마디에 쓸게 | bar 1 = IV |
| 그 다음, 두 번째 마디에는 딸림화음을 쓸게 | bar 2 = V |
| 딸림화음 다음에는 으뜸화음을 쓸게 | (V 다음 마디) = I |
| 버금딸림 화음 대신 다른 화음을 써보고 싶어 | IV → II (같은 SD 기능의 다른 화음) |
| 딸림화음을 첫 번째 마디에 사용하고 싶어 | bar 1 = V |

→ Claude 분석 — "Haiku가 이 맥락을 해석할 수 있나 / 하네스로 어디까지 가능한가":

해석 능력은 **충분하다 — 단, 자유 추론이 아니라 "제약된 선택"으로 설계할 때.**
난이도 4단계:
- **Tier 1 (Haiku 쉬움)** — 용어→화음 룩업 + 마디 슬롯 배정.
  프롬프트에 *교육과정 용어 ↔ 기능 ↔ 로마숫자 글로서리* + Q2 per-bar 스키마면 된다.
- **Tier 2 (기능 표 있으면 쉬움)** — 기능 기반 대안 선택 ("다른 화음" → IV↔II).
- **Tier 3 (규칙으로 가능, 품질은 few-shot 의존)** — 제안·가이드
  ("버금딸림 다음 뭐가 좋아?" → SD→D 규칙으로 V 제안).
- **Tier 4 (경계)** — 분위기 어휘("좀 더 슬프게")를 화성으로. Haiku가 *무언가*
  내놓지만 일관성↓ → 자유 생성 대신 **큐레이션 진행 중 분위기로 고르기**로 폴백.

핵심 — 언어 해석은 어렵지 않다. 진짜 설계 난점은 둘:
1. **상태가 있는 마디별 편집.** 예시가 전부 대화형이다("그 다음", "대신",
   "~하고 싶어"). chord 입력은 현재 진행 상태(Q2의 `Chord[]`)를 매 요청에 싣고
   학생 발화를 *델타*로 적용하는 stateful 흐름이어야 한다. 프롬프트가 아니라
   아키텍처 변경 — chord 레이어가 상태형 서브플로우가 된다.
2. **Haiku는 작은 모델.** 추론(Tier 3~4)을 시킬수록 분산이 커진다. chord 생성을
   "알려진 구조(글로서리·기능표·큐레이션 진행·per-bar 슬롯) 안의 선택·편집"으로
   가두면 실패 모드가 bounded된다. 자유 화성 생성은 피한다.

교육과정 레이어링: 중1 핵심은 주요 3화음(I·IV·V) → 글로서리도 I·IV·V를 코어로,
II·III·VI·VII는 "더 해보고 싶을 때"의 확장으로 둔다.
현실적 목표: **Tier 1·2 탄탄히 / Tier 3는 사용자 규칙을 가이드로 / Tier 4는
큐레이션 폴백.** (사용자도 "그 정도 분량 구현 불필요"라 명시)

#### melody (사용자 규칙 — "가장 주관적, 어법이 가장 어려움")
1. 기본적으로 **key에 종속**되는 음. **diatonic 7음**을 쓰는 게 자연스럽다.

사용자 추가 — **멜로디는 규칙으로 서술 불가.** 어떤 음을, 어떻게, 어떤 리듬으로
연주할지는 빅데이터급 데이터로 경향을 잡을 수는 있어도 현재로선 불가능.
**제약 안에서 랜덤으로 돌리는 편이 나을 수 있다.** 제약:
- **리듬**: 최소 단위 16분음표.
- **음역**: C3–C5 기준, ±5 semitone 정도의 여유. key에 따라 다르게 설정.
- **음 선택**: key의 diatonic 음 우선 (규칙 1).

→ Claude 메모:
- 이 "제약 랜덤" 방향은 Strudel과 잘 맞는다 — `.scale("C:major")` + `irand`/`choose`로
  스케일 도수를 랜덤 추출하고, 16-step 그리드에 `~`(쉼표)를 섞으면 제약 안의 멜로디가 된다.
- 아키텍처 함의: 멜로디는 **클라이언트 측 제약 랜덤**으로 생성 가능 — drum/bass/chord와
  달리 Haiku의 구조적 역할이 작다. Haiku는 "분위기 어휘 → 제약 조정"(음 밀도, 음역 폭,
  쉼표 비율) 정도로 축소되거나 생략 가능. 신뢰성·비용 면에서 오히려 이점.
- few-shot GOLD 예시는 Haiku를 거칠 때만 의미. 멜로디의 진짜 레버는 *제약의 경계값*이다.

### 2. 조성·화성 일관성 — `key`와 코드 진행을 thread
지금 첫 레이어가 bpm을 정한다. **key(조성)와 코드 진행도 위로 끌어올려** 정하고,
이후 레이어 응답에 함께 넘겨 모든 음(note) 레이어가 같은 조성·화성 위에 있게 한다.
- bass 규칙 (1)("chord의 root음 기반")이 성립하려면 bass 생성 시점에 코드 진행이
  *구조화된 데이터*로 있어야 한다 — 코드 문자열 파싱에 의존하면 깨지기 쉽다.
- 하네스 출력/입력 스키마에 `key` + `chordProgression` 필드 추가 검토. (Q2 참고)

### 3. Strudel 관용구 라이브러리를 프롬프트에 주입
하네스가 좋은 결과를 내는 Strudel 기법을 알아야 한다. 특히:
- 유클리드 리듬 `(3,8)`, `(5,8)` — 적은 코드로 그루브 있는 패턴. **레버리지 최상**
- `<a b>` 마디별 교대, `[a b]` 묶기, `~` 쉼표, `a*2` 반복
- `.slow()/.fast()`, `.every(n, f)`, `.gain()`, `.room()`
- 드럼 샘플 선택 `s("bd:3 sd:2")`

### 4. few-shot이 규칙보다 강하다 — GOLD 예시 병행
모델은 규칙 나열보다 좋은 예시를 더 잘 모방한다. 레이어 타입별로
**완성된 좋은 4마디 루프 예시 3~5개**를 넣고, 규칙은 그 예시를 뒷받침하는
형태로. (현재 하네스 v2에도 예시가 있으나 수가 적고 품질 기준이 낮음)

### 5. 장르 프리셋
학생이 vibe를 고르게 한다 — 힙합 / 하우스 / lo-fi / 팝 발라드.
장르마다 규칙팩을 둔다. 사용자의 드럼 규칙도 이미 장르 인식적이다
("4-beat 하우스 킥은 예외"). 규칙을 장르 축으로 정리하면 자연스럽다.

### 6. 프롬프트 평가 루프 — 개선을 측정 가능하게
"좋아졌다"를 감으로 판단하지 않으려면:
- 학생이 쓸 법한 입력 테스트셋 (20~30개)
- 출력 Strudel을 한 화면에서 빠르게 청취·비교하는 dev 페이지
프롬프트를 고칠 때마다 같은 셋을 돌려 A/B 청취.

### 7. 음악적 검증 (과투자 경계)
`isStrudelCodeSafe`는 위험 토큰만 막는다. 추가로 — 파싱 성공 여부,
드럼 레이어에 다운비트 킥 존재 여부 정도의 가벼운 체크 + 실패 시 1회 재생성.
단 일회성 수업 앱이므로 정교한 음악 분석까지는 과투자.

## Q2 설계 초안 — 생성 순서 / 화성 데이터 threading

> 위 「레이어 타입별 규칙 초안」의 강한 규칙(bass (1) chord root 기반, chord (3)
> 마디당 1코드, melody (1) key 종속) 안에서 설계함.

### 문제
bass 강한 규칙 (1)은 "chord의 root음 기반" → bass 생성 시점에 코드 진행이 있어야
한다. chord는 key에 의존. 그런데 현재는 첫 레이어가 `bpm`만 정하고 학생이 임의
순서로 레이어를 쌓는다 → bass를 chord보다 먼저 추가하면 근거가 없다.

### 설계: "곡 토대(Foundation)" 사전 스텝
레이어 루프에 들어가기 전, **곡 토대를 한 번 확정**한다. 학생의 전체 곡 묘사 →
하네스가 `{ bpm, key, chordProgression }` 반환 → 이후 모든 레이어 요청에 이 토대를
함께 넘긴다.

```
Foundation (bpm·key·chordProgression)   ← 무조건 먼저
   ├─→ drum    (bpm만)                  언제든
   ├─→ chord   (progression을 voicing·리듬·음색으로 렌더)
   ├─→ bass    (progression의 마디별 root + 드럼 킥 리듬)   drum 후 권장
   └─→ melody  (key + progression, 강박에 코드톤 착지)
```

강제되는 건 **Foundation이 맨 먼저**라는 것 하나. 그 외엔 토대가 key·진행을 실어
나르므로 어떤 순서로 쌓아도 화성이 일관된다. 유일한 소프트 규칙: bass는 drum 뒤
(규칙 2 "킥과 비슷한 리듬" 충족용).

### 데이터 구조 (핵심 산출물)
bass 규칙 (1)이 동작하려면 코드 진행이 문자열이 아니라 **마디별 구조화
데이터**여야 한다. 코드 문자열 파싱에 의존하면 깨지기 쉽다.

```ts
type Foundation = {
  bpm: number;                          // 60–140
  key: { tonic: string;                 // "C", "G", "F#" ...
         mode: "major" | "minor" };
  chordProgression: Chord[];            // 길이 4 (4마디 루프, chord 규칙 3)
};

type Chord = {
  bar: number;                          // 1–4, 이 코드가 차지하는 마디 (chord 규칙 3)
  roman: string;                        // 키 상대 도수 "I" "vi" "IV" "V" "ii" ...
  root: string;                         // 절대 음명 "C" "A" "F" "G" — bass 규칙 (1)의 직접 입력
  quality: "maj" | "min" | "dom7" | "min7" | "maj7";
  notes: string[];                      // voicing된 코드음 ["c3","e3","g3"] — chord 레이어용
};
```

필드 근거:
- `root` — **bass 규칙 (1)의 직접 입력.** bass는 마디별 `root`를 받아 옥타브 내려
  라인을 구성. Q2에서 가장 중요한 필드.
- `quality` — bass 규칙 (3) 우선순위의 "코드의 5음·코드음"을 알려면 필요.
  chord voicing에도 쓰임.
- `notes` — chord 레이어가 재유도 없이 바로 사용 + bass 규칙 (3) 3순위 "코드음"의 출처.
- `bar` — chord 규칙 (3) 마디당 1코드를 핀.
- `roman` — 키 상대 표기 → 이조(transpose)·가독성, 진행을 키와 분리해 재사용.

### 코드 진행 — 창작 판단으로 고른 큐레이션 (전수 열거 안 함)
12키 × N진행을 나열하지 않는다. 키 상대(로마 숫자)로 4마디·마디당 1코드 진행을
소수만 고른다. key가 정해지면 그 위에서 실현:

| # | 진행 | 성격 | 실현 예 |
|---|------|------|--------|
| 1 | I–V–vi–IV | 가장 익숙한 팝, 밝음 | C–G–Am–F |
| 2 | I–vi–IV–V | 클래식/복고 (50s) | C–Am–F–G |
| 3 | vi–IV–I–V | 살짝 애절하게 시작해 해결 | Am–F–C–G |
| 4 | I–vi–ii–V | 턴어라운드 — 사용자가 언급한 ii–V 진행 학습 | C–Am–Dm–G |
| 5 | i–VI–VII–i | 단조, K-pop/록에 흔함 | Am–F–G–Am |

1~4는 장조, 5는 단조 프레이밍. 더 필요하면 같은 기준으로 추가 — 핵심은
"소수 큐레이션". (chord 아이디어 (2) "프리셋"의 구체화)
각 진행을 왜 골랐는지의 근거(기능 흐름·SD–D–T·5도 하행)는
§1 「화성 기능」 참고.

### 남은 하위 질문
- Foundation 사전 스텝을 별도 화면 단계로 둘지, 첫 레이어 추가에 녹일지 (UX).
- chord 레이어를 학생이 여전히 "추가"하는 레이어로 둘지 — 진행은 토대에서 정해지고
  chord 레이어는 그 voicing/음색이므로, "코드 추가"보다 "반주 스타일 고르기"에 가깝다.
- 하네스 입력 스키마에 Foundation을 싣는 방법 (`apps/api`의 `LayerType` 흐름 변경 영향).
